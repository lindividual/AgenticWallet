1. 前端不能出现硬编码的文字，要支持多语言。
2. 项目是开发阶段，不需要考虑“兼容”和“最小改动”，应该考虑代码简洁合理，易于拓展维护
3. 布局和文字尽量用逻辑方向（start/end），不要写死 left/right
4. 文案以“清晰、简洁、可操作”为第一原则，不要为了显得热情、完整或高级而堆砌字数。
5. 删除不增加信息的 filler words。谨慎使用“轻松地 / 快速 / 简单 / 立即 / 哦不 / 抱歉 / 请 / 感谢 / ！”这类词；只有在它们真的传达能力差异、状态差异或操作后果时才保留。
6. 避免重复表达。标题、描述、按钮各自承担不同信息，不要换一种说法把同一件事重复三遍。
7. 指引型文案优先“先讲收益，再讲动作”。例如比起“输入手机号以接收更新”，更优先“为接收更新，请输入手机号”。
8. 错误、提醒、状态变化要直接说明发生了什么、影响是什么、下一步做什么；不要用玩笑化、撒娇式、过度拟人化的表达弱化问题。
9. 形容词和副词默认从严控制。像“最佳”“强大”“智能”“便捷”“友好”这类词，如果不能被具体功能或结果支撑，就不要写。
10. 同一个概念在全站只使用一个固定名称，避免混用近义词。按钮文案、字段名、导航名、agent 能力名都应维护统一词表。
11. 按钮文案优先使用明确动作，不用空泛词。优先“保存钱包”“查看行情”“继续支付”，少用“确认”“提交”“下一步”这类脱离上下文的词，除非该流程内语义已经非常明确。
12. 描述文案补充“为什么”而不是重复“做什么”。如果标题已经说明动作，正文应优先补充收益、条件、风险或结果。
13. 文案要考虑翻译和本地化，不依赖中文语气词、双关、网络梗，也不要依赖英文长度假设来设计布局。
14. 写完后通读一遍，优先删词，再判断是否还清楚；能少一句说明白，就不要多一句。
