# Agent Employee Implementation Goals

Date: 2026-03-15

## 1. 背景

`AgenticWallet` 依然是一个面向钱包用户的产品，不是一个面向企业管理者的后台系统。

我们的核心设想是：

- App 像一家持续营业的门店
- 用户的钱包、资产、关注列表、阅读与交易行为，像门店里的商品、客户与经营现场
- Agent 像门店里的员工，持续替用户整理、跟进、提醒、总结、优化

这个比喻主要用于内部产品设计。

用户不需要理解后台有多少个 Agent、它们如何协作，但可以明确知道产品里有一个专门的助手入口；只要它确实能提升体验，这种显式入口本身也是体验的一部分。

## 2. 核心目标

我们要实现的不是“让用户使用 Agent”，而是“让 App 自己把钱包打理得更好”。

目标拆解如下：

1. 让钱包体验更省心
   - 自动整理资产信息、重点内容、提醒与建议
   - 降低用户自己判断“下一步该看什么、做什么”的负担

2. 让钱包体验更及时
   - 在合适的时间给出摘要、提醒、风险提示和机会发现
   - 让用户更早看到与自己相关的变化，而不是靠主动搜索

3. 让钱包体验更懂用户
   - 根据持仓、关注、访问、阅读、交易与反馈逐步调整输出
   - 避免泛化、重复、无关的内容和打扰

4. 让 Agent 可持续优化
   - 用户不需要理解复杂的 Agent 机制
   - 也可以明确感知到有一个专门助手在服务自己
   - 但我们内部必须能观测 Agent 是否在正确地工作、是否真的改善了体验

## 3. 产品原则

### 3.1 用户侧可感知，但不增加理解负担

- 可以保留一个明确的助手入口，让用户知道钱包里有一个专门的助手
- 不要求用户理解多 Agent 架构
- 不要求用户手动配置“员工”或管理“团队”
- 用户感受到的重点仍然是体验变得更聪明、更省心，而不是后台有几个机器人

### 3.2 内部侧显式管理 Agent

- 每个 Agent 的职责、触发条件、输入上下文、输出动作、权限边界都必须明确
- 每次 Agent 运行都必须可回放、可审计、可评估
- Agent 不是 prompt 拼接物，而是一个受控的产品能力单元

### 3.3 先建议和整理，再逐步自动化

- 第一阶段重点是整理、摘要、提醒、推荐、优先级调整
- 涉及交易、转账、签名等高风险动作时，默认只建议，不自动执行
- 自动化能力必须建立在可解释和可观测之上

### 3.4 优化目标是体验和结果，不是“像人”

- 我们不追求强人格化的虚拟员工演出
- 我们追求的是少打扰、够及时、有依据、能持续变好

## 4. 目标中的 Agent 角色

第一阶段，Agent 应该围绕钱包用户的核心体验工作，而不是泛化成无边界助手。

### 4.1 资产整理 Agent

职责：

- 整理用户首页和资产视图的重点信息
- 发现值得提升曝光的资产、主题、卡片和动态
- 根据用户近期行为调整展示优先级

用户感知结果：

- 首页更清晰
- 重点资产与重点变化更靠前
- 信息噪音减少

### 4.2 内容与摘要 Agent

职责：

- 生成日报、专题、动态摘要
- 把市场变化与用户关注资产联系起来
- 在有限篇幅内给出真正相关的信息提炼

用户感知结果：

- 更容易快速看懂今天发生了什么
- 内容和自己的钱包状态更相关

### 4.3 提醒与跟进 Agent

职责：

- 在价格、波动、持仓、关注标的、页面停留等事件发生时触发提醒
- 判断是否值得提醒，避免过度打扰
- 在提醒中附带原因和上下文

用户感知结果：

- 提醒更及时
- 提醒更有用
- 垃圾推送更少

### 4.4 推荐与引导 Agent

职责：

- 基于用户持仓、浏览、阅读、收藏和交易线索生成下一步建议
- 在用户犹豫或停留时给出轻量引导
- 帮助用户更快发现相关资产、话题和功能入口

用户感知结果：

- 钱包“知道我下一步可能会关心什么”
- 推荐更贴近当前状态

## 5. 用户侧体验目标

用户可以知道产品里有一个专门助手，也可以在需要时主动打开它；与此同时，整体体验仍然应该稳定体现以下变化：

1. 打开 App 时，更快进入重点
2. 日报和动态摘要更贴近自己的钱包状态
3. 提醒更少但更准
4. 推荐更相关，重复更少
5. 钱包整体像被持续整理过，而不是静态工具集合

## 6. 钱包场景下的关键体验难点

区块链钱包相比普通内容或工具产品，存在更高的理解门槛、更强的风险感和更明显的错误焦虑。

内置 Agent 的重要价值之一，是帮助我们在不增加界面复杂度的前提下，补上这些体验缺口。

### 6.1 币和链的理解门槛高

典型问题：

- 用户需要理解资产属于哪条链
- 用户需要理解同名资产、不同链资产、原生币与代币的区别
- 在收款、转账、充值、提现等流程中，用户容易因链选择错误而紧张

设计目标：

- 主流程界面尽量简化，不把复杂链路知识一次性压给用户
- 在关键节点由 Agent 提供轻量解释、上下文提醒和风险确认

Agent 可发挥的作用：

- 在收款、转账等场景自动解释“当前资产是什么、所在链是什么、为什么这里必须选链”
- 在用户选择链、地址、资产时提示常见风险点
- 在不打断主流程的前提下，用更易懂的话把底层概念翻译成用户任务语言
- 在检测到高风险组合时提高提示强度，例如地址格式不匹配、链与资产不一致、用户历史上从未使用过该链

### 6.2 出错时用户容易焦虑

典型问题：

- 钱包功能依赖链上节点、索引服务、价格服务、签名状态和广播结果
- 只要延迟、失败、状态不明确，用户就容易担心“钱是不是没了”
- 普通错误文案无法有效安抚用户，也无法指导下一步动作

设计目标：

- 一旦发生错误、超时、待确认或状态异常，Agent 要第一时间介入
- 不只报错，还要解释当前状态、可能原因、风险边界和下一步建议

Agent 可发挥的作用：

- 在交易卡住、广播延迟、余额未刷新、签名失败等场景中主动介入说明
- 将技术错误翻译成用户能理解的状态语言，例如“交易正在链上等待确认，不代表资产丢失”
- 告诉用户当前最安全的下一步，而不是只给一个报错码
- 在长时间异常时持续同步进展，降低用户不确定感

### 6.3 其他可由内置 Agent 改善的体验点

1. 降低首次使用门槛
   - 帮助新用户理解钱包首页、资产页、收付款、网络切换等关键入口
   - 根据上下文做渐进式解释，而不是一次性做厚重教程

2. 降低操作前的心理负担
   - 在发送、授权、签名、切链等动作前，用简短说明帮助用户确认自己在做什么
   - 在高风险动作前补充结果预期和撤销边界

3. 缩短异常排查路径
   - 当用户遇到“为什么没到账”“为什么余额不对”“为什么价格波动异常”时，先由 Agent 做第一层归因解释
   - 减少用户在多个页面来回查状态

4. 减少通知与信息噪音
   - 不把所有市场变化都推给用户，而是结合持仓、关注和历史行为做筛选
   - 让提醒更像“有必要告诉你”，而不是“系统什么都发”

5. 提高复杂信息的可理解性
   - 把链上、市场、项目、风险这些复杂信息压缩成用户当前需要的那一层表达
   - 避免把专业术语直接堆给普通用户

6. 提高恢复体验
   - 在失败、超时、取消、重试等节点给用户更明确的心理预期
   - 帮助用户理解“现在系统正在做什么”和“我接下来该做什么”

7. 让产品更具连续性
   - 用户上一次看过什么、想做什么、做到哪一步，Agent 可以帮助接续上下文
   - 降低用户每次重新进入 App 都要重新建立心智状态的成本

## 7. 内部实现目标

为了让上面的体验成立，内部必须构建一层 Agent 运行与观测能力。

这层能力不是单独售卖的后台，而是 `AgenticWallet` 的内部操作层。

### 7.1 Agent 定义标准化

每个 Agent 至少要明确：

- `role`：它负责什么
- `trigger`：什么情况下运行
- `context`：它能读到哪些数据
- `decision`：它要产出什么判断
- `action`：它最终执行什么动作
- `permission`：哪些动作只能建议，哪些允许自动落地

### 7.2 Agent 运行链路标准化

每次运行都应该形成完整事件链路：

- `trigger`
- `agent`
- `context`
- `decision`
- `action`
- `result`
- `feedback`

这样后续才能进行回放、归因、A/B、调优和问题定位。

### 7.3 Agent 输出必须可评估

所有摘要、推荐、提醒、排序调整都不能只看“是否生成成功”，还要看：

- 是否被用户看到
- 是否被用户点击或采纳
- 是否造成负反馈
- 是否真的改善了体验指标或业务指标

## 8. 内部观测目标

我们必须把 Agent 当成一支数字化员工团队来观测和优化。

内部观测系统第一阶段至少覆盖以下维度：

### 8.1 运行状态

- 触发次数
- 成功率 / 失败率
- 平均耗时
- 重试情况
- 模型与外部依赖成本

### 8.2 输出质量

- 摘要是否空泛
- 推荐是否重复
- 提醒是否过多或无关
- 分类是否准确
- 是否命中已知错误模式

### 8.3 用户影响

- 日报打开率
- 内容点击率
- 提醒打开率
- 建议采纳率
- 页面引导交互率

### 8.4 业务结果

- 活跃度变化
- 留存变化
- 转化变化
- 重要功能使用率变化

### 8.5 安全与信任

- 是否出现误导性结论
- 是否越过权限边界
- 是否触发高频骚扰
- 是否在上下文不足时给出过强建议

## 9. MVP 范围

第一阶段不追求全自动智能钱包，而是先做一个能证明价值的闭环。

MVP 建议聚焦：

1. 每日摘要 / 日报
2. 事件驱动提醒
3. 首页信息优先级整理
4. 推荐卡片或引导卡片
5. 内部 Agent 运行流水和效果看板

MVP 成功的标志不是“Agent 很酷”，而是：

- 用户更常打开
- 用户更快找到重点
- 用户更常点击有价值内容
- 用户对提醒和推荐的负反馈下降

## 10. 非目标

当前阶段不把以下内容作为核心目标：

- 让用户显式配置多个 Agent
- 做强人格化陪伴式聊天体验
- 在没有观测和权限控制前开放高风险自动执行
- 为了拟人而拟人

## 11. 成功标准

这个方向成立，至少要满足以下判断：

1. 用户可以知道有一个专门助手存在，但更重要的是能明显感知钱包体验在持续变好
2. 内部可以追踪每个 Agent 的行为、质量、结果和问题
3. Agent 的输出对用户体验和业务指标产生可验证的正向影响
4. 整个系统具备继续扩展更多 Agent 能力的清晰边界与基础设施

## 12. 一句话定义

`AgenticWallet` 的目标不是为了展示 Agent 而展示 Agent，而是把钱包逐步变成一个会自己整理、提醒、总结和优化的产品；在此基础上，允许用户明确感知并使用一个专门助手入口，同时在内部建立一套可观测、可评估、可持续优化的 Agent 运行体系。
